# ClinDraft Backend

The backend service for **ClinDraft**, a modern healthcare application that provides a clinical AI assistant for drafting clinical notes, summarizing patient contexts, and managing medical data.

This backend is built on a **Single Source of Truth** architecture, unifying all clinical data (Conditions, Medications, Allergies, Observations) into a single structured patient context.

## 🚀 Tech Stack

- **Runtime:** Node.js
- **Language:** TypeScript
- **Framework:** Express.js
- **Database:** SQLite (via `better-sqlite3`)
- **ORM:** Drizzle ORM
- **Validation:** Zod
- **AI Integration:** Google Gemini API (`@google/generative-ai`)
- **Execution:** `tsx` for seamless TypeScript execution

## 📡 API Endpoints

The system is built on a streamlined 8-endpoint architecture:

### Patient Management
- `POST /api/patients` - Create a new patient profile.
- `GET /api/patients` - List all patients.
- `GET /api/patients/:id` - Get basic demographics for a specific patient.

### Clinical Context (Single Source of Truth)
- `GET /api/patients/:id/context` - Retrieve the full clinical context for a patient (Problems, Medications, Allergies, Observations).
- `PUT /api/patients/:id/context` - Update the entire clinical context. (Backend automatically generates UUIDs for any newly added items).

### Drafts & Notes
- `GET /api/patients/:id/drafts` - Get all clinical drafts/documents for a specific patient.
- `POST /api/drafts` - Save a new draft/document.
- `PUT /api/drafts/:id` - Update the content of an existing draft.

### AI Agent
- `POST /api/agent/chat` - Send a message to the AI clinical assistant (automatically fed with the patient's current clinical context).

## 🗄️ Data Model Explanation

Our architecture uses a **Single Source of Truth** model for clinical data to reduce relational complexity. The database consists of 3 main tables:

1. **`patients` table**: 
   * Stores standard demographic info (`firstName`, `lastName`, `dateOfBirth`, `gender`, `mrn`).
   * **The `context` column**: This is a powerful JSON column that acts as the single source of truth for the patient's medical history. It holds structured arrays for `activeProblems`, `currentMedications`, `allergies`, and `relevantObservations`.
2. **`drafts` table**: 
   * Stores clinical notes, summaries, and documents generated by the AI or user. Linked to the patient via `patientId`.
3. **`chat_messages` table**: 
   * Stores the chat history (Role: `user` or `agent`) between the provider and the Gemini AI assistant. Linked to the patient via `patientId`.

## 🧠 System Workflows

### How the Patient Context is Created
The Patient Context is the core data structure used by both the frontend UI and the AI Agent.
1. When a patient is queried (`GET /api/patients/:id/context`), the `ContextGenerator` fetches the patient's record.
2. It dynamically calculates the patient's age based on their Date of Birth and generates a `patientOneLiner` (e.g., *"44-year-old Male"*).
3. It merges this generated one-liner with the structured JSON `context` stored in the database. 
4. If the patient has no clinical data yet, it automatically returns a safe default structure with empty arrays for `activeProblems`, `currentMedications`, `allergies`, and `relevantObservations`.

### How the Agent/Chat Workflow Works
1. The frontend sends a user message and a `patientId` to `POST /api/agent/chat`.
2. The backend immediately logs the user's message into the `chat_messages` table.
3. The backend fetches the patient's **entire up-to-date clinical context** using the `ContextGenerator`.
4. The `PromptBuilder` constructs a strict **System Prompt** (which enforces rules like: *No diagnosing, no prescribing*) and a **User Prompt** (which injects the patient's active problems, meds, and allergies right above the user's actual question).
5. This heavily-contextualized prompt is sent to the **Google Gemini API**.
6. The AI's response is logged to the `chat_messages` table and returned to the frontend to be displayed.

## 🚫 Intentionally Out of Scope

To keep the application highly performant, maintainable, and focused on AI integration, the following architectural decisions were made to keep certain complexities out of scope:

1. **Relational Clinical Tables:** We intentionally avoided creating highly normalized SQL tables for every clinical concept (e.g., separate tables for `conditions`, `medications`, `allergies` with foreign keys). Instead, we use a **Single Source of Truth JSON column**. This eliminates complex SQL `JOIN`s, drastically reduces backend boilerplate, and makes it trivial to pass the entire medical history directly to the LLM.
2. **Authentication & Authorization:** JWTs, OAuth, and Role-Based Access Control (RBAC) are out of scope for this foundational build.
3. **Granular Audit Trails:** While we track basic `createdAt` and `updatedAt` timestamps, we do not currently maintain a historical version-control log of every edit made to the patient's context array.

## ⚠️ Product & Clinical Safety Risks Considered

Building AI for healthcare requires strict guardrails. The following risks were explicitly considered and mitigated:

1. **AI Hallucinations & Unsafe Medical Advice:** The largest risk is the LLM attempting to act as a doctor (diagnosing or prescribing). 
   * **Mitigation:** We implemented a two-layer defense. First, a keyword pre-filter (`isUnsupportedTask`) intercepts high-risk words (e.g., "diagnose", "prescribe") and blocks the request before it even reaches the LLM. Second, the `PromptBuilder` injects a strict system prompt heavily sandboxing the AI's behavior to a documentation assistant.
2. **Data Integrity & Sync Errors:** In a highly relational database, a network failure could result in a medication saving but the corresponding allergy failing, leading to dangerous incomplete records.
   * **Mitigation:** Transitioning to a **Single Source of Truth (JSON payload)** guarantees atomic updates. The entire context is either 100% saved or 100% rejected, preventing fragmented patient states.
3. **Cross-Patient Data Contamination:** 
   * **Mitigation:** Context is constructed entirely dynamically per-request inside the `AgentController`. There is no global state, meaning it is impossible for Patient A's context to accidentally leak into a prompt meant for Patient B.

## ⏱️ Trade-offs Made Due to Time Limits

To deliver a highly functional MVP quickly, the following trade-offs were made:

1. **JSON Column vs. SQL Normalized Tables:** 
   * *Trade-off:* While the JSON column makes reading/writing blazingly fast and simple, it makes deep database analytics much harder. For example, running a SQL query like *"Find all patients taking Lisinopril"* is significantly more difficult than it would be with a dedicated `medications` table.
2. **SQLite vs. PostgreSQL:**
   * *Trade-off:* SQLite was chosen for zero-configuration, lightning-fast local development. However, it lacks advanced JSON indexing capabilities and horizontal scalability. A real production EHR would require PostgreSQL.
3. **No Concurrent Edit Conflict Resolution:**
   * *Trade-off:* The `PUT /api/patients/:id/context` route currently blindly overwrites the database. If two doctors edit the same patient at the exact same time, the last one to click save will silently overwrite the other's changes. Implementing ETags or optimistic locking was skipped for speed.

## 🚀 Improvements for a Real Clinical Pilot

If this application were to transition from a prototype to a real-world clinical pilot, the following improvements would be strictly necessary:

1. **Database Migration:** Swap SQLite for **PostgreSQL**. PostgreSQL offers vastly superior scalability, true horizontal concurrency, and robust JSONB indexing capabilities needed for complex clinical analytics.
2. **Authentication & Authorization:** Implement strict OAuth/OIDC authentication and Role-Based Access Control (RBAC) to ensure only verified medical professionals can access PHI (Protected Health Information).
3. **Comprehensive Audit Logging:** Implement a HIPAA-compliant audit trail that records every single `read` and `write` access to a patient's record, tagged with the exact user ID and timestamp.
4. **Optimistic Locking for Concurrent Edits:** Implement ETags or version numbers on the `context` object to prevent two providers from accidentally overwriting each other's changes if they edit a chart simultaneously.
5. **Automated Testing:** Implement a full suite of unit and integration tests (e.g., using Jest/Supertest) integrated into a CI/CD pipeline to guarantee no regressions in API behavior.

## 🛑 What We Would Refuse to Ship (Even if Asked)

In healthcare software, establishing hard ethical and safety boundaries is just as important as writing good code. We would categorically refuse to ship the following:

1. **Autonomous Diagnostic AI:** We will not ship any feature that allows the AI to autonomously generate medical diagnoses, dictate treatment plans, or prescribe medications. The AI must strictly remain a "Human-in-the-Loop" documentation and summarization assistant.
2. **Unencrypted PHI Transmission:** We will refuse to deploy the application on any architecture that allows Protected Health Information to be transmitted over unencrypted HTTP, or stored at rest without enterprise-grade encryption.
3. **Shared / Generic Accounts:** We will refuse to implement "clinic-wide" generic logins. In clinical software, every action must be tied to a distinct, auditable human identity for liability and safety tracking.

## 🐛 Troubleshooting

**"fetch failed" error from Gemini API (`/api/agent/chat`):**
If you encounter a `TypeError: fetch failed` when hitting the Gemini API locally or in production, this is typically caused by Node.js attempting to use IPv6 to resolve the Google API endpoint on a network that doesn't fully support it, or due to a general DNS resolution timeout. 
*Fix:* You can force Node.js to use IPv4 by setting the environment variable `NODE_OPTIONS=--dns-result-order=ipv4first` before starting the server.
